MÓDULO 02 · DATOS EMBEBIDOS

Configuración embebida: el JSON dentro del HTML

La herramienta no tiene una base de datos, ni un servidor, ni un archivo de configuración aparte. Toda la configuración —los formularios, las listas de opciones, las reglas— vive como un bloque de JSON dentro del propio HTML. Es el truco que hace que un solo archivo baste para todo.

📋 JSON ~10 min de lectura Nivel: principiante
1El truco del script type=application/json

Un <script> que el navegador NO ejecuta

Normalmente, <script> ejecuta JavaScript. Pero si le pones type="application/json", el navegador dice: "esto no es JavaScript, no lo ejecuto". Lo deja como texto dentro del DOM. Y desde JavaScript puedes leerlo con document.getElementById(...).textContent y usar JSON.parse() para convertirlo en un objeto.

HTML
<!-- Este script NO se ejecuta. El navegador lo guarda como texto. -->
<script type="application/json" id="vanillaConfig">
{
  "version": 2,
  "data": { ... },
  "schema": { ... }
}
</script>

<!-- Y este script SÍ se ejecuta y lee el de arriba: -->
<script>
const el = document.getElementById("vanillaConfig");
const config = JSON.parse(el.textContent);
console.log(config.version); // 2
</script>

¿Por qué no usar un archivo .json aparte con fetch?

Porque fetch no funciona en file:// (sin servidor). El navegador bloquea fetch de archivos locales por seguridad (CORS). Con el JSON embebido en un <script>, no hay petición de red: el dato ya está en el DOM desde que se carga la página. Cero-red, cero-problemas.

2La estructura del JSON: data + schema

El JSON embebido tiene dos grandes secciones: data (los datos que la herramienta usa) y schema (la definición de los formularios).

📦 data
Listas de opciones, mapas, reglas globales. Ej: regions: ["NA","EU","LATAM"], mediaTypes: ["Vid","Pic"], kw_count: "5".
📐 schema
Definición de los formularios: qué secciones tienen, qué campos, qué validaciones. El JS lee esto y construye todo el formulario dinámicamente.
JSON simplificado
{
  "version": 2,
  "data": {
    "lists": {
      "regions": ["NA", "EU", "APAC", "LATAM", "BR"],
      "platforms": ["PC", "PS5", "Xbox", "Switch"]
    },
    "rules": {
      "kw_count": "5",
      "media_fmt": "{type}_{key}_{platform}_{cl}_(1)"
    }
  },
  "schema": {
    "forms": [
      {
        "id": "bug",
        "label": "Bug",
        "sections": [ ... ]
      }
    ]
  }
}
3Separar datos de código

El schema es la "fuente de verdad"

No hay código JavaScript que diga "crea un campo llamado Team". En cambio, el JSON dice { "id": "team", "label": "Team", "type": "text" } y el JavaScript tiene una función genérica que lee esa definición y crea el campo. Los datos describen qué; el código describe cómo.

¿Por qué es tan potente?

1

El equipo de Database edita el JSON, no el código

Quien mantiene las listas (regiones, mapas, POIs) solo cambia datos en la pestaña Data o importa un CSV. No toca JavaScript.

2

"Exportar herramienta" reescribe el JSON

Cuando alguien hace cambios y pulsa "Exportar herramienta", se descarga un nuevo bug_tool.html con el JSON actualizado embebido. Quien lo reciba verá esos cambios automáticamente.

3

El código no cambia nunca, solo los datos

El JavaScript es el mismo para todos. Lo que cambia entre versiones es el bloque vanillaConfig. Es como un programa y su archivo de configuración — pero en un solo archivo.

4Tu primer momento

Pulsa Ver el JSON para leer el bloque embebido en esta misma página y mostrarlo formateado. Es exactamente lo que hace la herramienta al arrancar.

JavaScript 
// Leer el JSON embebido y mostrarlo formateado:
const el = document.getElementById("vanillaConfig");
const config = JSON.parse(el.textContent);
console.log(JSON.stringify(config, null, 2));
