MÓDULO 00 · BIENVENIDA

¿Qué es la Bug Report Tool y cómo funciona?

Vamos a estudiar una aplicación real, construida en un solo archivo HTML, que sirve para reportar bugs. No necesitas instalar nada: la abres en el navegador y funciona. En este curso la desmontamos pieza por pieza para que entiendas cómo se construye.

~8 min de lectura Sin instalar nada HTML + CSS + JS
1Qué es la herramienta

Un solo archivo que lo hace todo

La Bug Report Tool es una herramienta para que un equipo de QA reporte bugs de forma estructurada. Vive en un único archivo .html. No usa servidores, no necesita conexión a internet, no requiere instalar nada. La copias a una carpeta compartida, la abres con doble clic y funciona.

Esto parece magia, pero no lo es. Toda la información que la herramienta necesita —los formularios, las listas de opciones, las reglas de validación, los nombres de archivos de media— está embebida dentro del propio HTML como un bloque de JSON. El JavaScript de la página lee ese JSON y construye la interfaz a partir de él.

Las tres capas en un solo archivo

La herramienta usa las tres tecnologías fundamentales de la web, todas dentro del mismo .html:

HTML — Estructura
Los contenedores, las pestañas, los campos. El esqueleto de la página.
CSS — Estilo
Colores, posiciones, animaciones. Cómo se ve todo: el tema claro, los bordes, el layout.
JS — Comportamiento
La lógica: leer el JSON, validar campos, generar el output, guardar en localStorage.
2La restricción de diseño

Cero-red, cero-install, file:// compatible

Estas tres restricciones definen todo el diseño de la herramienta:

1

Cero-red

No usa fetch, no carga Google Fonts, no importa librerías de un CDN. Todo está dentro del archivo. Funciona sin conexión a internet.

2

Cero-install

No hay npm install, no hay bundler, no hay Node.js. La abres con file:// y ya está. La pueden usar personas que no saben instalar nada.

3

Distribución simple

Se copia a una carpeta compartida (un sharefolder). Cada persona en el equipo la abre desde ahí. Los cambios se guardan en el navegador de cada uno con localStorage.

3Recorrer la herramienta

Antes de empezar a construir, familiarízate con lo que ya existe. Abre bug_tool.html en tu navegador (doble clic sobre el archivo) y explora estas zonas:

1

Barra superior (pestañas)

Arriba: las pestañas Bug, Regression, Data y Rules. A la derecha, botones para exportar CSVs, guardar sesión y exportar la herramienta completa.

2

Barra de tiles

Debajo de la barra superior: pequeños chips de colores que cambian según lo que escribas. Rojo si falta un campo obligatorio, verde si todo está listo.

3

Formulario (izquierda)

El panel donde el tester escribe los datos del bug: equipo, región, plataforma, descripción, coordenadas, pasos para reproducir, etc.

4

Output (derecha)

El texto que se genera automáticamente a partir de lo que escribiste. Lo copias y lo pegas en el issue tracker.

5

Panel de media (abajo)

Genera nombres de archivos de media (vídeo, fotos, logs) con una regla configurable.

6

Pestaña Data

Editas las listas de opciones (regiones, plataformas, mapas, POIs) y las reglas globales.

7

Pestaña Rules

Editas las validaciones de cada campo y las condiciones entre campos.

4Las tres capas en acción

5Tu primer ejercicio

Abre la herramienta y llena un bug

Antes de seguir, haz esto. Es la mejor forma de entender por qué la herramienta existe y qué hace.

1

Abre bug_tool.html

Doble clic sobre el archivo. Se abre en tu navegador. No necesita servidor.

2

Llena el formulario Bug

Escribe algo en cada campo: Team, Región, Plataforma, Mapa, descripción, coordenadas con el formato (X, Y, Z) (X, Y, Z), build URL con getbuild://, etc.

3

Mira la barra de tiles

Los chips cambian de rojo a verde según llenas campos. La tile verde "✓ Sin errores" aparece cuando todo está completo.

4

Copia el output

Pulsa "Copiar todo" en el panel derecho. Pega el resultado en un editor de texto. Ese es el texto que pegarías en el issue tracker.
