MÓDULO 03 · SCHEMA

Formularios y tipos de campo en el schema

El JSON del schema define cada campo del formulario: su tipo, su etiqueta, sus validaciones y cómo se transforma al output. En este módulo recorremos los 5 tipos de campo y todas las propiedades que controlan su comportamiento.

📋 Schema ~12 min de lectura Nivel: intermedio
1Los 5 tipos de campo

Cada campo del schema tiene una propiedad type que determina qué widget se renderiza y cómo se comporta:

text
Texto de una línea. El más común: Team, Región, Coordenadas.
textarea
Texto multi-línea. Descripciones largas, pasos para reproducir.
autocomplete
Texto + dropdown filtrable. Las opciones salen de una lista de Data (source).
keywords
Chips de palabras clave. Valida cantidad exacta, unicidad y no-overlap con el cuerpo del bug.
mirror
Copia de otro campo. No aparece en el formulario, solo en el output (ej: CCS aparece 2 veces).
JSON — Ejemplos de cada tipo
// text — una línea
{ "id": "team", "label": "Team", "type": "text", "required": true }

// textarea — multi-línea
{ "id": "description", "label": "Descripción larga", "type": "textarea", "rows": 4 }

// autocomplete — texto + dropdown
{ "id": "region", "label": "Región", "type": "autocomplete", "source": "regions" }

// keywords — chips
{ "id": "keywords", "label": "Palabras clave", "type": "keywords", "kwCount": "@kw_count", "kwOverlap": true }

// mirror — copia para el output
{ "id": "ccsMirror", "label": "CCS (copia)", "type": "mirror", "source": "ccs" }
2Propiedades de campo

Cada campo puede tener estas propiedades (todas opcionales excepto id y label):

3Secciones y modos de output

Los campos se agrupan en secciones. Cada sección tiene un mode que controla cómo se combinan sus campos en el output:

JSON — Sección modo "joined"
{
  "id": "resumen",
  "label": "Resumen (encabezado)",
  "mode": "joined",        // todo en una línea
  "sep": " - ",            // separador entre campos
  "fields": [ ... ]
}
JSON — Sección modo "lines"
{
  "id": "notas",
  "label": "Notas",
  "mode": "lines",           // cada campo en su línea
  "heading": "+Notas:+",        // texto antes del bloque
  "fields": [ ... ]
}

¿Cuándo usar cada modo?

joined: Para encabezados tipo resumen, donde varios campos van en una sola línea separados por guiones. Ej: Team Alpha - NA - MP - Highrise.

lines: Para bloques de información donde cada campo merece su propia línea. Ej: pasos numerados, notas, resultados.

4Plantillas con {value}

La propiedad template controla cómo se transforma el valor del campo al output. {value} se reemplaza por lo que el tester escribió:

Ejemplos de plantillas
// El valor tal cual:
"{value}"  →  Escribe "foo"  →  foo

// Con prefijo:
"CL: {value}"  →  Escribe "12345"  →  CL: 12345

// Con formato wiki:
"* *Perfil:* {value}"  →  Escribe "user"  →  * *Perfil:* user

// Pasos numerados (con perLine: true):
"# {value}"  + perLine  →  Escribe "Abrir menu\nCerrar app"  →
# Abrir menu
# Cerrar app

// Con paréntesis:
"({value})"  →  Escribe "MSS"  →  (MSS)

// Con encabezado de sección:
"+ {value}"  →  Escribe "variable1"  →  + variable1

Pulsa Ver los 5 tipos para ver las definiciones JSON de cada tipo de campo formateadas.

JavaScript 
// Las 5 definiciones de tipo de campo:
const types = [
  { type: "text",         label: "Texto (una línea)" },
  { type: "textarea",     label: "Texto (varias líneas)" },
  { type: "autocomplete", label: "Texto + dropdown" },
  { type: "keywords",     label: "Palabras clave (chips)" },
  { type: "mirror",       label: "Copia de otro campo" }
];
console.log(JSON.stringify(types, null, 2));
